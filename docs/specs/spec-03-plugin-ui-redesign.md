# spec-03 — UI/UX редизайн плагина NN FabKit

**Статус:** черновик 2026-04-24, ждёт уточнения scope (см. §0).
**Привязка к ADR:** ADR-002 (DC-UX), ADR-007 (RU-локализация). Может породить ADR-019 о выборе HtmlDialog vs UI.inputbox.
**Применимо к:** оболочка `NN::FabKit` + ветка `MetalFab`. Шаблон расширяемый под `MebelFab`.

---

## 0. Открытый вопрос: scope «дизайн»

Заказчик-0 сказал «придумать и реализовать дизайн NN-Kit плагина SketchUp» — три возможные интерпретации:

| | Что входит | Кто делает |
|---|---|---|
| **A. UI/UX плагина** | HtmlDialog панели, layout, флоу команд, формы создания/редактирования, properties panel, toolbar. Заменяем messagebox-driven UI на современный. | Я (Ruby + HTML/CSS/JS). |
| **B. Visual brand** | Логотип NN FabKit, цветовая схема, иконки для меню и toolbar, splash. | Дизайнер / графический инструмент (могу описать концепт, но не нарисовать качественно). |
| **C. И то и другое** | A + B одним sprint. | A — я, B — отдельный загруз дизайнером или AI-генератором. |

**Default scope этого spec — A** (UI/UX, без графики). B спросим у заказчика отдельно. Если ответ «нужно и B» — добавляется sprint с генерацией icon set + логотипа через Figma/AI tool, отдельный документ.

## 1. Текущее состояние UI

| Команда | Сейчас | Проблема |
|---|---|---|
| Создать «Профильная труба» | `UI.inputbox` 3 поля + dropdowns | Нет preview сечения, нет фильтра по сортаменту, нет недавних значений. На каталоге 62 типоразмера — листать долго. |
| Экспорт IGES | `UI.savepanel` | OK. |

**Главный недостаток:** «Создать «Профильная труба»» — самая частая команда — самая неудобная. На production-моделях с десятками труб разных типоразмеров пользователь будет дёргать её много раз; узкий inputbox без preview быстро надоедает.

## 2. Цель

Дать заказчику-0 **боковую панель** в SketchUp («NN FabKit Inspector»), всегда открытую при работе с плагином, через которую идут все команды. Заменить inputbox-driven workflow на постоянную панель с:
- Сортаментом (поиск, фильтр) и быстрой кнопкой «Создать».
- Properties выделенного компонента (что это, типоразмер, длина, материал) с кнопкой «Перегенерировать» и «Экспорт IGES».
- Статусом MCP-сервера (после spec-02).

## 3. Технология

**`UI::HtmlDialog`** — родной механизм SketchUp для embedded HTML/CSS/JS окон. Поддерживается с SU 2017+.

Ключевое:
- HTML/CSS/JS живут в `plugin-sketchup/src/nn_fabkit/ui/html/`.
- Two-way bridge:
  - `dialog.add_action_callback("name") { |dlg, *args| ... }` — JS → Ruby.
  - `dialog.execute_script("...")` — Ruby → JS.
- Можно сделать dock'ed (привязать к боковой панели SU).
- Persistent размер и позиция (`Sketchup.read_default`).

**Стек фронтенда:** простой — vanilla JS + минимальный CSS. Без React/Vue/build step.
- Минимизирует deployment complexity (всё в `.rbz`, ничего не собирать).
- Не загружает SU тяжёлым runtime'ом.
- Если позже понадобится React — нарастим, но MVP — vanilla.

Альтернатива — Web Components с native browser API (пока не нужно).

### 4.2. Modal: «Изменить параметры трубы»

При клике «Изменить параметры…» — модальный HtmlDialog с теми же полями (сечение, длина, марка), предзаполненный текущими значениями. Кнопки «Применить», «Применить как новый компонент», «Отмена».

Это решает проблему «как поменять трубу» без полноценного DC-EntityObserver (block 5.5 spec-01) — можно сделать **раньше**, чем observer.

### 4.3. Toolbar (опционально)

`UI::Toolbar` с иконками для самых частых команд:
- 🔧 Создать трубу
- 📤 IGES экспорт
- 🔍 Открыть Inspector

Иконки 24x24 PNG (требование SU). Если идём в B-scope (visual design) — нарисуем; иначе — placeholders text-based.

## 5. UX-флоу

### 5.1. Создание новой трубы

**Сейчас:** Меню → диалог 3 поля → OK → труба в [0,0,0]. 4 действия.

**После:** Inspector открыт всегда. Найти типоразмер (поиск + фильтр) → длина/марка уже введены прошлый раз → кнопка «Создать» → труба в [0,0,0]. **2 действия** + меньше когнитивной нагрузки (видишь весь сортамент).

### 5.2. Редактирование выделенной трубы

**Сейчас:** Невозможно через UI (только Component Options readonly).

**После:** Выделить → панель показывает свойства → «Изменить параметры…» → diалог с предзаполненным → «Применить». **3 действия**, без боли.

### 5.3. Экспорт IGES

**Сейчас:** Выделить → меню глубоко в подменю → savepanel. 3 действия + поиск меню.

**После:** Выделить → кнопка «Экспорт в IGES» в Inspector → savepanel. **2 действия**, кнопка всегда на виду.

## 8. План реализации

**Sprint A — каркас:**
- [ ] `Inspector.rb` — открывает HtmlDialog, dock right.
- [ ] Главное меню → пункт «Открыть Inspector».
- [ ] Базовый HTML/CSS layout + sections (header, sortament, selection).
- [ ] Загрузка каталога труб через JSON (Ruby читает `metalfab/catalogs/gost-...json`, передаёт в JS через `execute_script`).

**Sprint B — функционал create:**
- [ ] Search + фильтры по сортаменту.
- [ ] Кнопка «Создать» → JS отправляет JSON-RPC-стиль команду в Ruby через `add_action_callback`.

- [ ] Last-used значения (длина, марка) сохраняются через `Sketchup.write_default`.

**Sprint C — Selection panel:**
- [ ] `Sketchup::SelectionObserver` подписан → при изменении пушит state в JS.
- [ ] JS показывает свойства выделенной трубы или «не выделено».
- [ ] Кнопка «Изменить параметры…» → модальный TubeModal.
- [ ] Кнопка «Экспорт в IGES…» → вызывает существующий `ExportIges.call`.

**Sprint D — Toolbar:**
- [ ] `UI::Toolbar` с 3 иконками.
- [ ] Иконки PNG 24x24, размещение в `ui/html/assets/toolbar/`.

## 9. Зависимости

- Этот spec **не блокирован** ничем — можно стартовать параллельно spec-01 и spec-02.
- Spec-02 (MCP мост) **усилит** этот spec — Inspector сможет показывать статус MCP-сервера и работать как панель управления для разработчика.
- При наличии MCP-моста (spec-02) тестирование Inspector ускорится в разы — `eval_ruby` позволит вызывать любые команды панели программно.

## 10. Открытые вопросы

1. **Scope A или A+B?** (см. §0) — это первое, что нужно уточнить у заказчика-0.
2. **Dock side** — справа (как Outliner) или плавающее окно? По умолчанию — справа.
3. **HtmlDialog vs WebDialog** — HtmlDialog современный, поддерживается SU 2017+. WebDialog — legacy. Идём на HtmlDialog.
4. **Реактивность Selection.** SelectionObserver триггерится много (при каждом клике в модели). Дебаунс 100мс перед обновлением UI, иначе flicker.
5. **Тёмная тема.** SU 2024+ имеет dark mode — Inspector должен подхватывать через `prefers-color-scheme` CSS media query или явный API. Проверить что в SU 2025 это работает.
