# NN FabKit — Конечная цель продукта

**Статус:** канон. Пересмотр — отдельный ADR со ссылкой на этот документ.
**Зафиксировано:** 2026-04-26 (заказчик-0).

---

## Зачем существует продукт

**NN FabKit — экосистема для проектирования и production-поставки металлоконструкций и корпусной мебели поверх SketchUp Pro 2021+.**

С момента когда заказчик-конструктор открывает модель, до момента когда станок получает корректные данные для резки/раскроя — **минимальное время и минимальное ручное вмешательство**.

## Три выхода продукта

### 1. Данные для лазерного трубореза

**Формат:** `.igs` (IGES 5.3, BREP) для Friendess CypTube; `.dxf` для альтернативных CAM.

**Что генерит:**
- Профильная труба с любыми резами:
  - Perpendicular cut (стандартный торец)
  - **Mitre под произвольным углом** (для угловых стыков, рамок, перегородок)
  - Сквозные отверстия (через стенку)
  - Пазы / notches (для штифтов, болтов, T-образных соединений)
- Поддерживаемые профили: квадрат / прямоугольник (далее — круглая, уголок, швеллер)

**Принцип:** вход — DC-компонент в SketchUp с метаданными в `nn_metalfab` attribute_dictionary; выход — готовый файл, читаемый CAM **без ручной переработки**.

### 2. Раскроечные данные для форматного станка по дереву

**Формат:** `.dxf` или формат специфичный для OpenCutList (OCL) Cutting Diagrams.

**Что генерит:**
- ЛДСП, фанера, ЛДВП, плита
- Кромка (ABS, ПВХ, меламин) — наклейка по периметру или выборочно
- Сверление под штифты, минификс, eccentric
- Фрезеровка под петли, направляющие

**Принцип:** интеграция с OCL **только по конвенции имён материалов** (см. ADR-016) — наш плагин не импортирует GPLv3-код OCL, не пишет OCL-словарь на материалы. Свой раскрой строим параллельно по тем же принципам.

### 3. Техническая документация

**Формат:** LayOut (`.layout` SU Pro), `.pdf` экспорт.

**Что генерит:**
- Спецификация деталей с типоразмерами и количествами (cut-list)
- Чертежи отдельных деталей с размерами и отметками cut'ов
- Спецификация метизов / фурнитуры
- Технологические карты сборки (порядок операций, инструмент, материалы)

**Принцип:** генерация шаблонная, через LayOut Ruby API; пользователь правит финальное оформление в LayOut, но не делает рутинную часть.

## Ключевая UX-фича: One-click cut-list

Это **convergence point** продукта — точка, в которой плагин превращает 3D-модель в production-данные одним действием.

### Workflow

1. Заказчик-конструктор моделит конструкцию (перегородку, раму, корпус, шкаф) в SketchUp из параметрических компонентов плагина
2. Жмёт **одну кнопку** в Inspector / toolbar
3. Плагин:
   - Сканирует выделение или всю модель
   - Извлекает все отдельные детали с поворотами, размерами, материалами, **резами**
   - **Группирует одинаковые** (с указанием количества):
     - «3× труба 40×20×2 L=600 mitre 45° оба конца»
     - «12× ЛДСП 600×400×16 кромка ABS по периметру»
     - «8× винт DIN 7991 M8×30»
   - Раскладывает их **физически в SU** (рядом с моделью или в отдельном слое) для визуальной проверки
   - Готовит выход: `.igs`/`.dxf` для станков, `.pdf` LayOut для документации

### Зачем именно «one click»

Заказчик-0 — мебельщик + металлист — делает это вручную. На production-модели с 50+ труб и 30+ ЛДСП-деталей рутинная работа извлечения cut-list'а съедает часы. Цель плагина — **минуты**.

## Принципы реализации

### Точная BREP-геометрия (не «примерно 3D-блок»)

LOD-1/LOD-2 со скруглениями, mitre, отверстиями, пазами. Каждая деталь после применения cut'ов — production-grade геометрия, не визуализация. Уровни детализации зафиксированы в [ADR-014](knowledge-base/09-architecture-decisions.md).

### Метаданные на каждой детали

`nn_metalfab` / `nn_mebelfab` attribute_dictionary хранят на компоненте:
- Типоразмер (например `40x20x2`), профиль, ГОСТ
- Размеры, толщина стенки, внешний радиус скругления
- Материал (марка стали для металла, материал плиты для дерева)
- **Состояние резов** (`cut_z0_angle_deg`, `cut_zL_angle_deg`, holes, slots)
- Масса погонная, calc-helpers

Cut-list-сборщик читает эти метаданные → точный production output. См. [ADR-005](knowledge-base/09-architecture-decisions.md).

### Локальный offline pipeline

Никаких cloud-зависимостей. Всё работает под лицензией заказчика на SketchUp на его машине. Manifest-обновления плагина — единственная сетевая активность (опциональная). См. [ADR-008](knowledge-base/09-architecture-decisions.md).

### MCP-мост Claude ⇄ SketchUp (опционально)

Заказчик может в любой момент попросить Claude через Project Knowledge / MCP сгенерировать новый компонент из текстового описания: «труба 60×30×2 длиной 800 мм с миттер-резом на обоих концах под 45°» → плагин получает команду через TCP, создаёт DC. См. [spec-02](specs/spec-02-mcp-bridge.md).
