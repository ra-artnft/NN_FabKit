# 05. OpenCutList — референс для проектирования

> **Статус (22.04.2026): референс, не интеграция.** Согласно ADR-016 этот документ перешёл из роли «как интегрироваться с OCL» в роль «как устроен OCL и чем нам это полезно при проектировании собственной системы». OCL у заказчика остаётся как отдельный инструмент, но наш продукт **полностью заменяет** ту часть workflow, которую сейчас делает OCL (спецификация, раскрой, подсчёт материала). Мы **не пишем OCL-атрибуты** в свои компоненты, **не зависим** от его конвенций и **не включаем** его код. Соглашения, которые выглядели как «контракт с OCL» (имя материала, длина по X), сохранены — но уже как **наши собственные** решения, совпадающие с OCL случайно (и удобно).

## Что такое OCL

OpenCutList (OCL) — open-source (GPLv3) плагин для SketchUp, разработан проектом L'Air du Bois. Изначально для столярки, де-факто используется и для металлоконструкций, мебели, МДФ, пластиков. Генерирует:
- список деталей (Parts List) с размерами и количеством
- карты раскроя плит (Cutting Diagrams) — для Sheet Goods
- лейблы для наклейки на детали
- стоимость и вес
- экспорт в CSV / PDF

У заказчика OCL **уже установлен и используется сейчас**. В целевой картине его место займёт наш продукт (см. ADR-016). Разбор ниже — про то, как OCL устроен внутри и что из этого полезно забрать себе как идею (не как код).

## Лицензия — критично

GPLv3 (copyleft). Это накладывает жёсткое ограничение: **мы не можем включать OCL-код внутрь нашего плагина**, иначе наш плагин обязан стать GPLv3 целиком (что несовместимо с коммерческим закрытым продуктом).

Безопасный способ интеграции — **через стандартные механизмы SketchUp**, без импорта OCL-модулей:
- мы пишем **материалы и атрибуты компонентов**, которые OCL читает сам
- мы не вызываем OCL-методы напрямую из нашего кода
- мы можем вдохновляться архитектурой OCL (public data model, published paper), но не копировать код

Это не «обходной путь», это именно тот контракт, который OCL предоставляет миру — добавлять совместимый контент через атрибуты.

## Что OCL читает у компонента

Это ключевая часть интеграции.

OCL смотрит на каждый `ComponentInstance` (не Group — OCL игнорирует Groups как концептуально «не детали»), извлекает:

1. **Bounding Box** в локальных осях компонента → длина, ширина, толщина.
2. **Имя definition** → основа для группировки «одинаковые детали».
3. **Материал** (top-level material), назначенный либо на инстанс, либо на definition, либо на её faces → определяет тип (Solid Wood / Sheet Good / Dimensional / Hardware / ...).
4. **Имя инстанса** (`instance.name`) — опциональная метка конкретной копии.
5. **OCL-атрибуты**, если они присутствуют (словари с префиксом `lairdubois_opencutlist_*`) — тонкая настройка для конкретной detali.

## Типы материалов OCL

Материал в OCL — это `Sketchup::Material` + дополнительные атрибуты в словаре материала. Тип выбирается в UI OCL при настройке материала.

| Тип | Русский | Когда применять к нашим компонентам |
|---|---|---|
| **Solid Wood** | Массив дерева | Редко, для деталей из массива |
| **Sheet Good** | Листовой материал | **Листы металла, ДСП, фанера** |
| **Dimensional** | Пиломатериал стандартного сечения | **Профильная труба, уголок, швеллер, круглый прокат** — основной тип для нашего металла |
| **Hardware** | Крепёж и метизы | Саморезы, болты, сварочные швы как «элементы» |
| **Edge Banding** | Кромка для плитных | Обычно не для металла |
| **Veneer** | Шпон | Только для мебели |

**Правило для нашего генератора:**
- профильная труба, уголок, швеллер, круглый прокат → **Dimensional**
- лист металла, ДСП, фанера → **Sheet Good**
- крепёж → **Hardware**
- сварочный шов (если моделируется) → **Hardware** или просто не в OCL

## Colour-hack заказчика — что реально происходит

В дампах Профильная_труба и Лист у заказчика виден материал «Труба профильная» (цвет RGB 64,64,64) и «Лист» (цвет RGB 86,86,86). Разные компоненты с **одинаковым именем материала** будут объединены OCL в одну строку спецификации (потому что OCL группирует по `material.name`, плюс по `definition.name`).

То есть «передача информации через цвет» работает не через цвет как таковой, а через **уникальность материала**: каждый типоразмер/вид проката = отдельный материал, и цвет просто помогает **визуально** их различать в 3D-окне. OCL читает **имя материала** (цвет ему безразличен).

Это означает два следствия:

1. Нам надо создавать **отдельный материал на каждый типоразмер**. Имя материала — это и есть «идентификатор для OCL».
2. Цвет можем выбирать мы (детерминированно из типоразмера — например, hash типоразмера → палитра серых оттенков), или заказчик через настройки — не критично.

## Схема именования материалов

Предложение, которое унифицирует UX для заказчика и нужды OCL:

```
Схема: <Русский_тип> <typesize> <марка>
Примеры:
  "Труба 40x20x2 Ст3сп"
  "Уголок 50x50x5 Ст3сп"
  "Лист 1500x3000x2 Ст3пс"
  "Швеллер 10П Ст3сп"
  "Труба 89x3 09Г2С"          — для круглой трубы
```

Каждый материал:
- **Имя** по схеме (это и есть OCL-группировка)
- **Тип** в OCL настраивается один раз (Dimensional для трубы/профиля, Sheet Good для листа)
- **Цвет** — серый оттенок, детерминированно от hash(typesize)
- **Description** (опционально) — полное обозначение с ГОСТ: `"Труба 40x20x2 ГОСТ 30245-2003 / Ст3сп ГОСТ 27772-88"`

## Программное создание материала

```ruby
def self.ensure_material(model, profile_type, typesize, grade)
  name = build_material_name(profile_type, typesize, grade)
  mat = model.materials[name]
  return mat if mat

  mat = model.materials.add(name)
  mat.color = color_for(profile_type, typesize)    # серый детерминированный
  mat.alpha = 1.0
  # Устанавливаем OCL-атрибуты (словарь на Material)
  #   Имена ключей подлежат проверке по актуальному исходному коду OCL;
  #   см. src/ladb_opencutlist/ruby/model/material_attributes.rb
  ocl_dict = "lairdubois_opencutlist_material_attributes"
  mat.set_attribute(ocl_dict, "type", ocl_type(profile_type))
  #   0 = UNDEFINED, 1 = SOLID_WOOD, 2 = SHEET_GOOD,
  #   3 = DIMENSIONAL, 4 = HARDWARE, 5 = EDGE_BANDING, 6 = VENEER
  mat
end

def self.ocl_type(profile_type)
  case profile_type
  when :rect_tube, :square_tube, :round_tube, :angle, :channel, :ibeam
    3  # DIMENSIONAL
  when :sheet, :plate
    2  # SHEET_GOOD
  when :bolt, :screw, :weld
    4  # HARDWARE
  else
    0  # UNDEFINED
  end
end

def self.build_material_name(profile_type, typesize, grade)
  prefix = case profile_type
           when :rect_tube, :square_tube then "Труба"
           when :round_tube              then "Труба"
           when :angle                   then "Уголок"
           when :channel                 then "Швеллер"
           when :ibeam                   then "Двутавр"
           when :sheet, :plate           then "Лист"
           else profile_type.to_s.capitalize
           end
  "#{prefix} #{typesize} #{grade}"
end
```

**Замечание:** точные ключи атрибутов OCL следует сверять с актуальными исходниками OCL (они менялись между v5 и v7). Основной источник истины — `src/ladb_opencutlist/ruby/model/material_attributes.rb` в репозитории `lairdubois/lairdubois-opencutlist-sketchup-extension`. Перед релизом нашего плагина — сверить.

## Локальные оси — критично для OCL

OCL интерпретирует bounding box компонента в **его локальной системе координат**. Для корректных результатов:

- Для **Dimensional** (труба/профиль): длина → ось X, поперечные габариты → Y и Z. То есть компонент расположен «вдоль красной оси».
- Для **Sheet Good** (лист): длина → X, ширина → Y, толщина → Z.

У заказчика в `Профильная_труба.skp` мы видим: truба 25×25×1000, длина вдоль оси **Z**, а не X. Это нестандарт для OCL (если включён «Automatic orientation», OCL сам определит — наибольшее измерение = длина, наименьшее = толщина). Но при выключенной опции — ошибка.

**Правило для нашего генератора**: ставим оси так, чтобы длина была по X, независимо от того, включена ли auto-orient. Это снимает всю неопределённость.

```ruby
# Переориентация: свой компонент всегда рисуем длиной по X
def self.place_axes_for_tube(defn, length_mm, width_mm, height_mm)
  # Создаём сечение в плоскости YZ, потом Follow Me вдоль X
  entities = defn.entities
  cross_section = draw_tube_cross_section(entities, width_mm, height_mm, wall_mm, radius_mm)
  path_edge = entities.add_line([0, 0, 0], [length_mm.mm, 0, 0])
  cross_section.followme(path_edge)
  path_edge.erase!
end
```

## Solid Components

OCL требует, чтобы компонент был «solid» (замкнутое тело без дыр и зазоров), иначе определение толщины/объёма работает некорректно. Для наших профилей это автоматически, если рисуем через Follow Me с замкнутым сечением. Проверка:

```ruby
defn.entities.grep(Sketchup::Face)         # все faces
# Solid если group/defn отвечает:
defn.manifold?    # true если solid; false — где-то дыра
```

## Что OCL НЕ читает у компонента

Чтобы не ожидать лишнего:
- **Не читает** наш словарь `nn_fabkit` — он его не знает и игнорирует.
- **Не читает** `dynamic_attributes` напрямую — только финальную геометрию после применения DC-scale.
- **Не читает** GUID — только имена.

Это значит: мы можем хранить в `nn_fabkit` любые метаданные, OCL их проигнорирует. И наоборот — для OCL важно только имя компонента, имя материала, bounding box.

## Cutting Diagrams для нашего листа

Для Sheet Good OCL умеет генерить карты раскроя — это и есть что нужно для листового металла. Параметры, которые OCL берёт:
- Толщина листа → из настройки материала (он запоминает, какие толщины доступны)
- Размеры сырого листа (stock sheet) → из настройки материала
- Детали и их размеры → из компонентов с этим материалом

В UI OCL заказчик:
1. Настраивает материал «Лист 1500x3000x2 Ст3пс» → тип Sheet Good, толщина 2 мм, стандартные листы 1500×3000
2. Запускает Cutting Diagrams → получает схему раскроя листа и список обрезков.

Наша задача — правильно сгенерировать компоненты с правильным материалом. Дальше OCL сам.

## Роль нашего плагина в связке с OCL

Кратко:
- **Генерим компонент с правильной геометрией, осями, именем, материалом**.
- **Не трогаем OCL-атрибуты материала** в базовом сценарии (заказчик сам настраивает материалы в OCL один раз). Опционально — умеем подставлять дефолтные настройки, если материал только что создан.
- **Не включаем код OCL**.
- **Не дублируем функционал OCL**: не пишем свой раскрой плит, не пишем свои cut lists. Если нужно — делаем отдельный экспорт (DXF для станка), параллельно OCL-раскрою.

## Проверка интеграции

После генерации компонента и вставки в модель, в OCL:
1. Вкладка **Parts List** → компонент виден, сгруппирован по имени definition, с размерами.
2. Вкладка **Materials** → материал виден, тип установлен.
3. Если тип Sheet Good и настроены размеры стока → вкладка **Cutting Diagrams** строит схему.

Если в OCL компонент не появляется — типичные причины:
- Group вместо Component
- Пустое имя definition
- Материал не назначен (или назначен с типом UNDEFINED)
- Нулевой объём (плоский компонент, не solid)
- Ошибка в локальных осях (редко — OCL сам ругается)
