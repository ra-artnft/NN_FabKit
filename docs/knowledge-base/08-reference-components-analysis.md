# 08. Разбор референсных компонентов заказчика

## Контекст

Заказчик прислал два `.skp` файла со своими рабочими компонентами:
- `Профильная труба.skp` — DC, заготовка квадратной/прямоугольной трубы
- `Лист.skp` — DC, заготовка листового металла

Обе модели — SketchUp 2025 (25.0.575), Windows. Пути: `C:\Users\Заурядный\Desktop\Металл\`. Единицы — миллиметры.

Файлы были проанализированы через собственный Ruby-дампер SkpDump v1.0 (см. `skp_dumper.rb`). Ниже — результаты с критикой и списком доработок.

## `Профильная труба` — структура

### DC с двумя механизмами выбора типоразмера

Definition имеет `_formatversion: 1.0` и `_hasbehaviors: 1` — это полноценный DC, построенный, видимо, поверх семплового DC из Components Sampler (см. раздел «Мусор» ниже).

**Видимые пользователю параметры (Component Options):**
- `square` — «Квадрат» — dropdown типоразмеров квадратной трубы
- `rectangle` — скрытый dropdown прямоугольных (только два варианта)
- `LenX` (Глубина), `LenY` (Ширина), `LenZ` (Длина), `RotX/Y/Z`

**Формулы:**
```
_lenx_formula: "square/10"
_leny_formula: "square/10"
_lenz_formula:  (нет — свободный ввод)
```

`square` хранится как строка в сантиметровом выражении (см. `_lenx_error: "<span class=subformula-success>25</span>/10"`), поэтому делится на 10 для получения сантиметров. Странный костыль, объясняемый историей изменения единиц в проекте, но работает.

### Геометрия — идеализированная коробка

```
entities_summary: { Edge: 12, Face: 6 }
```

Это **прямоугольный параллелепипед без радиусов и без стенки**. Размеры достигаются через **scale-transformation** на инстансе:

```
transformation = [2.5, 2.5, 1.0, ...]     # матрица scale по X, Y, Z
```

Базовая геометрия в definition — 10×10×1000 мм. Инстанс скейлится до 25×25×1000. Edges и faces при этом не перестраиваются — движок DC просто растягивает.

### Палитра типоразмеров

`_square_options` (URL-decoded):
```
10, 15, 20, 25, 30, 40, 50, 60, 80, 100, 120, 140, 150, 160, 180, 250, "не квадрат"=0
```

`_rectangle_options` — **только две записи**: `10x20` и `10x30`. Остальные типоразмеры прямоугольной трубы (20x40, 40x60, 60x80 и т.д.) **отсутствуют**.

### Мусор в файле

В `model.definitions` присутствует компонент `Sumele` с описанием:
> «Sumele is the Senior Strategic Marketing Manager at SketchUp. When she's not taking new products to launch, you can find her sharing deep belly laughs with her husband...»

Это семпловый DC из папки `C:/ProgramData/SketchUp/SketchUp 2021/SketchUp/Components/Components Sampler/`. Видимо, заказчик использовал его как основу для своего DC (общая структура атрибутов `_iscollapsed`, `_inst_y_*`, `_name`, `_scaletool_*` характерна для DC-семплов), а потом стёр инстанс, но definition и связанные материалы остались.

**Материалы-мусор:**
```
Sumele_Hair, Sumele_Dress3, Sumele_Dress, Sumele_Skin,
Sumele_Shoe, Sumele_Jewelry, Sumele_Dress2
```

В модели из одной трубы присутствуют 7 чужих материалов. В реальных проектах такого мусора будет больше.

**Что с этим делать в продукте:** наш плагин при открытии проекта предлагает `purge_unused` (см. 02-idioms). Делаем это опционально, с подтверждением пользователя, чтобы не удалить то, что он считает «я ещё использую».

### Рассогласование кэша

На инстансе в служебном словаре с пустым именем `""`:
```
_last_lenx: 0.9842519685039371    (= 25 мм, совпадает с фактическим)
_last_leny: 0.984251968503937     (= 25 мм, совпадает)
_last_lenz: 39.37007874015748     (= 1000 мм, совпадает)
```

В этой модели кэш синхронизирован, но это частая точка отказа при ручных манипуляциях. Наш плагин должен уметь детектить расхождение.

## `Лист` — структура

### DC с толщиной из списка

**Параметры:**
- `LenX` (Длина) — TEXTBOX, свободный ввод
- `LenY` (Ширина) — TEXTBOX
- `LenZ` (Толщина) — LIST с опциями 1.5, 2, 3, 4, 5, 6, 8, 10, 12 мм

**Формул нет.** Каждый параметр пользователь задаёт сам.

### Геометрия

Те же 12 edges, 6 faces — плоский box. Размеры через scale:
```
transformation = [1.0, 666.66..., 0.0015, ...]
```

Здесь scale X = 1 (база 1000 → результат 1000 мм = 1 м), scale Y = 666 (база 1.5 мм → 1000 мм; это странно, но работает как DC), scale Z = 0.0015 (база 1000 → 1.5 мм).

### Опечатка

```
_lenz_formlabel: "Толшина"
```

Должно быть «Толщина». Опечатка у заказчика, видна в его Component Options.

### Рассинхронизация кэша в листе

```
_last_lenx: 39.37007874015748    (= 1000 мм, ок)
_last_leny: 39.37007874015748    (= 1000 мм, ок)
_last_lenz: 39.37007874015748    (= 1000 мм!!!)
```

В кэше `_last_lenz` — **1000 мм**, хотя фактическая толщина — 1.5 мм. Типичный DC-баг: значение в кэше не обновилось при перегенерации. Пользователь этого не замечает (UI показывает актуальное `lenz = 1.5 мм` через `0.05905... дюйма`), но любой наблюдатель, ориентирующийся на `_last_lenz`, получит ошибочные данные.

## Критика — что именно нужно починить и дополнить

### Геометрия

1. **Нет радиуса гиба у трубы.** Угол сечения — прямой. Для сварки, реза под 45°, программы трубореза — это критичная ошибка (см. ранее обсуждавшееся).
2. **Нет толщины стенки у трубы.** Сечение — сплошной прямоугольник, а не кольцо.
3. **Нет реальной толщины материала у листа.** Есть scale-трансформация, но не реальная геометрия «двухсторонней пластины с фасками» (для плазмы под фаской нужна геометрия).

### Метаданные

4. **Нет марки стали** ни у трубы, ни у листа. Для расчёта массы, оформления ТЗ, пометок на чертежах — нужно.
5. **Нет ГОСТ** в атрибутах. ГОСТ подразумевается, но не зафиксирован в data.
6. **Нет NC-метаданных** — каких отверстий, каких фасок, какой угол реза нужен.
7. **Нет обозначения профиля по ГОСТ** (маркировка вида `40x20x2 ГОСТ 30245-2003/Ст3сп ГОСТ 27772-88`).

### UX и данные

8. **Список прямоугольных размеров трубы обрезан** — только две записи. В стандарте десятки.
9. **Опечатка «Толшина» в листе.**
10. **Отсутствует единая система осей** — длина трубы по Z, а не по X (OCL может неверно интерпретировать, если Automatic orientation выключен).
11. **Чужие материалы (Sumele)** — мусор, который надо уметь убирать.
12. **Кэш `_last_*` рассогласован на листе** — надо валидировать при открытии.

## План доработки — сначала для трубы

Когда строим новый параметрический компонент трубы:

1. **Локальные оси:** длина вдоль X, ширина вдоль Y, высота вдоль Z.
2. **Сечение** — замкнутый контур с радиусами на углах:
   - внешний контур: прямоугольник width×height со скруглениями радиуса R
   - внутренний контур: прямоугольник (width−2×wall)×(height−2×wall) со скруглениями радиуса (R−wall)
   - face с дыркой посередине (через `Face#loops`)
3. **Экструзия через Follow Me** вдоль пути X длиной `length_mm`.
4. **DC-атрибуты** (сохраняем привычный UX заказчика):
   - `square` / `rectangle` → полный сортамент
   - `lenx/leny/lenz` → геометрические параметры
   - `rotx/y/z` → повороты
5. **Наши атрибуты в `nn_fabkit`:**
   - `profile_type: "rect_tube"`
   - `gost: "30245-2003"`
   - `typesize: "40x20x2"`
   - `width_mm`, `height_mm`, `wall_mm`, `outer_radius_mm`
   - `steel_grade`, `mass_per_m_kg`
   - `nc_ready: true/false` — флаг «готов к NC-экспорту»
6. **Имя компонента** — каноническое: `"Труба 40x20x2"`.
7. **Материал** — автоматически создаётся по схеме `"Труба 40x20x2 Ст3сп"` с типом OCL `dimensional`.
8. **Перерисовка при изменении DC** — через `EntityObserver` → полная регенерация геометрии.

## План доработки — для листа

Упрощённее:

1. **Оси:** длина X, ширина Y, толщина Z.
2. **Геометрия** — параллелепипед через `pushpull` (толщина как реальный Z-размер, не scale).
3. **DC-атрибуты** — как было, но с исправлением опечатки: `"Толщина"`.
4. **Наши атрибуты в `nn_fabkit`:**
   - `profile_type: "sheet"`
   - `gost: "19903-2015"` (горячекатаный) или `"19904-90"` (холоднокатаный)
   - `typesize: "1500x3000x2"`
   - `length_mm`, `width_mm`, `thickness_mm`, `steel_grade`
5. **Имя** — `"Лист 1500x3000x2"`.
6. **Материал** — `"Лист 1500x3000x2 Ст3пс"` с типом OCL `sheet_good`.

## Референс сохраняем

Эти два файла — ценный артефакт: показывают реальный workflow заказчика. Храним их в проекте:
- как образец для **регрессионного тестирования** (после нашей перегенерации — убедиться, что DC-UI остался узнаваемым)
- как источник для **миграции существующих проектов** — когда откроется старый `.skp` с «наивными» компонентами, плагин должен уметь их **распознать и заменить** на параметрически правильные
