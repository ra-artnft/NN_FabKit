# 09. Архитектурные решения (ADR)

Здесь фиксируются принципиальные решения, принятые в ходе обсуждений. Формат — короткий: контекст → решение → альтернативы → последствия. Номера ADR не меняются, записи не удаляются — если решение пересматривается, добавляется новый ADR со ссылкой на предшественника.

---

## ADR-001. Форк `mhyrr/sketchup-mcp` как основа исполнителя

**Контекст.** Нужен мост SketchUp ↔ Claude: плагин с TCP-сервером в SketchUp + MCP-сервер наружу. Писать с нуля — недели работы.

**Решение.** Форк `mhyrr/sketchup-mcp` (178⭐, MIT). Сохраняем TCP-архитектуру и базовый набор tools (`get_scene_info`, `create_component`, `transform_component`, `set_material`, `eval_ruby`). Расширяем:
- LayOut-tools (генерация .layout)
- Sortament-tools (вставка из каталога по GOST/typesize)
- Параметрический генератор как замена «create_component из примитивов»
- ImageRep feedback-loop

**Альтернативы.**
- Свой с нуля — дороже на 2–3 недели, без выигрыша.
- Использовать готовый `bearnetwork-brnkc` (китайский форк) — та же база, но менее активный, русификация не готова.

**Последствия.**
- Лицензия наша — MIT-совместимая с форком.
- Привязка к TCP-протоколу mhyrr как starting point. Возможность расширить/переписать протокол в v2.

## ADR-002. Dynamic Components как UX, наш плагин перерисовывает геометрию

**Контекст.** Заказчик использует DC с правым кликом → Component Options. Формулы DC умеют масштабировать box, но не умеют нарисовать сложное сечение с радиусами/стенкой. Для точных моделей (пригодных к NC) box-pretender не работает.

**Решение.** Сохраняем DC-интерфейс (`dynamic_attributes`, `lenx/leny/lenz`, `square`/`rectangle`, правокликовое меню) как UX-поверхность. Но при каждом изменении атрибутов наш плагин **полностью перерисовывает внутреннюю геометрию definition**: очищает entities, рисует корректное сечение, делает Follow Me, применяет материал.

**Альтернативы.**
- Чистые формулы DC (без нашего кода) — невозможно: формулы не меняют топологию.
- Свой UI полностью поверх DC — теряем знакомый workflow заказчика.
- Dynamic Components + LayOut как UI — излишне сложно, не даёт preview.

**Последствия.**
- EntityObserver на инстансы → ловим изменения → регенерация с дебаунсом.
- Сортамент живёт в JSON-файлах (ADR-004), не в DC-опциях.
- Наш плагин — обязательная зависимость: без него эти компоненты становятся обычными (но корректно отображаемыми) геометрическими телами.

## ADR-003. OCL интеграция через имя материала, без включения GPLv3-кода

> **Статус: SUPERSEDED (22.04.2026) → ADR-016.** Решение пересмотрено: OCL перестаёт быть целевой интеграцией и остаётся только референсом для проектирования собственной структуры учёта. Запись сохранена как исторический контекст.

**Контекст.** OpenCutList — GPLv3, используется заказчиком. Нельзя включать код OCL в наш продукт (иначе продукт должен стать GPLv3). Заказчик передаёт семантику материала через **имя материала** (неофициально называлось «через цвет», но реально OCL группирует по имени; цвет — только визуальный маркер).

**Решение.** Интегрируемся через публичный API SketchUp — создаём и присваиваем материалы с каноническими именами. OCL сам читает их, группирует, генерит Parts List. Мы в OCL-код не лезем.

Схема имён материалов:
```
<Тип> <typesize> <марка>
примеры:
  Труба 40x20x2 Ст3сп
  Уголок 50x50x5 Ст3сп
  Лист 1500x3000x2 Ст3пс
```

Тип OCL-материала (`solid_wood`/`sheet_good`/`dimensional`/`hardware`) — настраивает заказчик один раз в UI OCL (или мы пред-заполняем через атрибут `lairdubois_opencutlist_material_attributes.type`, если подтвердим точные ключи по актуальным исходникам OCL).

**Альтернативы.**
- Писать свой раскрой и cut-list — дублируем OCL, теряем зрелый функционал.
- Форкать OCL — заражаемся GPLv3.
- Вызывать OCL через TCP/local API — OCL такого API не даёт.

**Последствия.**
- Каждый типоразмер → отдельный Material.
- Оси компонента согласуем с OCL-конвенцией (длина по X).
- При открытии проекта наш плагин может проверить, что материалы имеют правильный OCL-тип; если нет — подсветить пользователю.

## ADR-004. Сортамент как JSON-файлы в `catalog/`, не в DC и не в коде

**Контекст.** Типоразмеров десятки, марок стали десятки. Держать в DC-опциях — уродство (`_square_options: "&10=...&15=...&..."`). Держать в коде — требует пересборки для добавления записи.

**Решение.** Каталог в `catalog/*.json` с устойчивой схемой (см. 06-sortament-ontology). Плагин читает при запуске и при выборе компонента. Пользователь может добавлять JSON без перезапуска.

**Альтернативы.**
- SQLite — оверхэд для read-only справочника.
- YAML — менее читается в IDE, больше проблем с UTF-8.
- CSV — нет структуры (params vs derived vs source_urls).

**Последствия.**
- JSON-схема версионируется (`$schema_version: "1.0"`).
- Ссылки на первоисточники (ГОСТ docs) в каждом файле.
- Минимум валидации при загрузке, fail-fast с понятной ошибкой.

## ADR-005. Свой attribute dictionary `nn_fabkit` параллельно DC

**Контекст.** На компоненте есть DC-атрибуты (`dynamic_attributes`), OCL-атрибуты (`lairdubois_opencutlist_*`), SketchUp-встроенные (`SU_DefinitionSet`, `SU_InstanceSet`). Наши метаданные (GOST, марка стали, NC-готовность, масса) не вписываются ни в одно место — их не должно быть в DC (иначе засоряем UX), и они не нужны OCL.

**Решение.** Собственный словарь `nn_fabkit` на definition (справочные данные: тип, ГОСТ, параметры сечения, масса погонного метра) и на instance (instance-specific: марка стали выбранная для этой копии, NC-флаги, пометки).

Ключи — snake_case, значения — примитивы (Integer, Float, String, Boolean). Никаких вложенных структур (SketchUp не поддерживает).

**Альтернативы.**
- Складывать всё в `dynamic_attributes` — засоряет UI Component Options.
- Использовать `SU_DefinitionSet.Price/Size/Url` — они предназначены для Extension Warehouse публикации, не для нашей онтологии.

**Последствия.**
- Уникальный namespace — не пересечётся с чужими.
- При экспорте (NC-module, LayOut, Reports) читаем всё из `nn_fabkit`.
- Сериализация в JSON тривиальна.

## ADR-006. Два Claude Projects: dev и product (одобренный план)

**Контекст.** В одном Claude Project — и разработческий контекст (мы строим сервис), и продуктовый контекст (сервис в работе с пользователем). Смешивать — системный промт получается кашеобразным.

**Решение.** Два отдельных Claude Project с шарящейся базой знаний.

**Dev-Project «Интеграция Claude ↔ SketchUp/LayOut — разработка»:**
- база знаний (эти файлы)
- системный промт «помогаешь планировать архитектуру, писать ТЗ для Claude Code, разбирать API, делать дизайн-решения»
- контекст со всеми нашими обсуждениями, roadmap, ADR

**Product-Project «NN FabKit — AI-проектировщик»:**
- та же база знаний (скопирована / ссылка)
- системный промт «ведёшь диалог с пользователем-мебельщиком/сварщиком, уточняешь параметры, генерируешь через MCP, оформляешь ТЗ»
- контекст короче, ориентирован на стандартные сценарии

Альтернативный путь (переключаемые режимы в одном проекте) → отложен до MVP.

**Альтернативы.** Один проект с режимами по триггеру — рассматривался, отклонён из-за размытости системного промта.

**Последствия.**
- При обновлении базы знаний — синхронизируем между двумя проектами вручную (до появления shared-project-files).
- Разделение «думать про продукт» vs «быть продуктом».

*Замечание: это предварительное решение. Пересмотр не исключён — пользователь обещал вернуться к вопросу о Claude Projects позже.*

## ADR-007. Локализация — русский

**Контекст.** Заказчик в России, интерфейс SketchUp в русской локали, документы ТЗ на русском, OCL русифицирован.

**Решение.** Весь пользовательский текст (меню, диалоги, сообщения, автотекст в LayOut, аннотации в моделях, имена компонентов и материалов) — по-русски. Код, идентификаторы, профильные типы в JSON, имена словарей атрибутов — по-английски (snake_case).

Для публичного релиза (этап 3) заложим i18n через `Sketchup::LanguageHandler` + отдельные `ru.strings` / `en.strings` файлы, но в MVP — монолит RU.

**Альтернативы.** RU+EN с первого дня — удваивает работу ради будущего, без пользы сейчас.

**Последствия.**
- Шрифты: проверить кириллицу в LayOut-шаблонах (стандартные работают, но кастомные могут не поддерживать).
- Клавиатурные шорткаты выбираем без русскоязычных символов.
- Сообщения в Ruby Console — по-русски для пользователя, для Claude-отладки — тоже по-русски, с префиксом `[NN::FabKit]`.

## ADR-008. Три production-экспортёра: OCL / LayOut ТЗ / NC

**Контекст.** Модель в SketchUp — исходник. На выходе нужны артефакты для производства. Объём и природа экспортёров — разные.

**Решение.** Три независимых модуля:

1. **OCL-экспортёр** — не пишем код, пользуемся существующим OCL. Наша ответственность — правильно размечать компоненты (материалы, оси, геометрия), чтобы OCL корректно генерил Parts List и Cutting Diagrams. См. 05.

2. **LayOut-экспортёр** — пишем. Генерируем `.layout` с шаблоном ГОСТ (рамка, штамп, автотекст), видами модели, размерами, спецификацией. Экспорт в PDF (для согласования с клиентом) и DWG/DXF (для передачи в CAD). См. 04.

3. **NC-экспортёр** — пишем. Для листового проката — DXF (поддерживается всеми лазерно/плазменными станками с 2D-сечением). Для объёмных изделий (металлоконструкций под труборез) — IGES или STEP через обходные пути (см. ADR-009).

**Альтернативы.** Единый экспортёр «всё в одно» — делает зависимость между форматами, усложняет отладку.

**Последствия.**
- Порядок реализации: OCL-разметка → LayOut-ТЗ → DXF → IGES/STEP.
- Каждый экспортёр — отдельный модуль плагина, с отдельным тестированием.

## ADR-009. NC-экспорт в IGES — через посредника (SolidWorks у заказчика)

> **Статус: SUPERSEDED (22.04.2026) → ADR-017.** Решение пересмотрено: SolidWorks-посредник убирается из цепочки с самого начала, собственный конвертёр входит в MVP. Запись сохранена как исторический контекст.

**Контекст.** SketchUp нативно не экспортирует IGES. Заказчик сейчас получает IGES так: сам рисует макет в SketchUp → сторонний исполнитель делает полную перерисовку в SolidWorks → SolidWorks экспортирует в IGES.

**Решение (MVP).** На этапах 0–2 **не меняем** эту цепочку. Наша задача — **повысить качество SketchUp-макета** до уровня, пригодного для NC (реальные радиусы, правильные метаданные). SolidWorks-посредник остаётся, но делает меньше работы — только перерисовку под CAD-формат, без «додумывания».

**Решение (этап 3+).** Убираем SolidWorks из цепочки. Варианты (исследуем позже):
- STEP через open-source Ruby-библиотеку (STEP более современен и поддерживается плазменными ЧПУ не хуже IGES, иногда лучше)
- прямая запись IGES через собственный конвертёр на базе OpenCASCADE (сложно, много работы)
- интеграция со сторонним SketchUp-плагином экспорта IGES (SimLab — платный, но возможный)
- предварительный экспорт SketchUp → DAE → сторонний open-source конвертер → IGES/STEP

**Альтернативы для MVP.** Сразу писать свой IGES-конвертёр → месяцы работы, отодвигает полезный релиз.

**Последствия.**
- Заказчик продолжает использовать SolidWorks, но **исполнитель-перерисовщик в SolidWorks получает от нас гораздо более чистый исходник** — с радиусами, с метаданными. Это сокращает его работу и снижает стоимость.
- Уточнить у заказчика, какая конкретно модель трубореза и какие форматы она ест — может быть сразу DXF/STEP достаточно.

## ADR-010. Feedback-loop через ImageRep — в MVP

**Контекст.** Диалог с Claude становится полноценным только если Claude видит результат генерации. Иначе это «слепая генерация» — код есть, ошибки в геометрии пропускаются.

**Решение.** Плагин после каждого значимого изменения модели (вставка компонента, изменение параметров, групповая перестройка) рендерит превью через `Sketchup::ImageRep`, отправляет Claude через MCP как image-input. Claude использует его для проверки и следующего шага диалога.

Параметры рендера:
- разрешение 1024×768 по умолчанию, настраивается
- текущий вид (active_view), без принудительной смены камеры
- опциональный second render с альтернативным углом (перспектива + ортогональный спереди)

**Альтернативы.**
- Textual dump через MCP — теряет пространственную интуицию.
- Многостадийный feedback (сгенерить, пользователь проверяет глазом, сообщает Claude результат) — разрывает диалог.

**Последствия.**
- Требует SketchUp 2018+ (ImageRep API), что у нас и так есть.
- Дополнительный трафик в MCP (image bytes), но не критично.
- В UI нашего плагина — галочка «отключить feedback-loop» на случай больших моделей.

## ADR-011. Локальное хранение state — гибрид

**Контекст.** Один мастер — много проектов, память живёт на оборудовании пользователя. Разные данные имеют разный scope.

**Решение.**

Глобальные данные пользователя — в пользовательском AppData:
- Windows: `%APPDATA%\NN FabKit\`
- macOS: `~/Library/Application Support/NN FabKit/`

Содержимое:
- `preferences.json` — UI-настройки, выбранный по умолчанию сортамент, марка стали
- `project-index.json` — список открывавшихся проектов, quick access
- `catalog/` — пользовательский оверрайд каталога (дополнение к встроенному)

Проектные данные — **рядом с `.skp`-файлом**, в одноимённой папке-сателлите:
- `<project-name>.nn/` — папка-сателлит
  - `project.json` — метаданные проекта (имя, заказчик, revision, статус)
  - `dialogue.md` — лог диалогов с Claude
  - `generated/` — промежуточные артефакты (DXF-ы, LayOut-ы, PDF-ы)
  - `references/` — референсные материалы, загруженные пользователем

Преимущество: мастер пересылает коллеге проект → копирует `.skp` + `.nn/` папку → вся история переносится.

**Альтернативы.**
- Всё в AppData с ссылкой на .skp — ломается при переименовании/перемещении .skp.
- Всё в проекте — глобальные настройки дублируются в каждом проекте.
- База данных SQLite в AppData — overkill для текстовых настроек.

**Последствия.**
- AppObserver на `onOpenModel` читает `<project-name>.nn/project.json` рядом с загруженным .skp.
- `.nn/` папка — текстовая, читается глазами, хранится в Git (если мастер ведёт версионирование).

## ADR-012. Colour-hack — сохраняем как legacy, но заменяем смыслом

**Контекст.** В моделях заказчика цвет материала используется как семантика (передача типа проката). Технически это работает только потому, что цвет привязан к уникальному имени материала — и именно имя читает OCL.

**Решение.** Не трогаем цветовую схему заказчика на существующих моделях. Для новых компонентов, генерируемых нашим плагином, — используем **детерминированный цвет**, вычисляемый из hash(typesize): серая палитра для стали, коричневая для дерева, голубая для алюминия. Цвет — визуальный маркер, не «носитель информации».

Если заказчик хочет переопределить цвет — не запрещаем. Но семантика (тип, марка, размеры) всегда живёт в имени материала + в атрибутах `nn_fabkit`.

**Альтернативы.**
- Заставить заказчика перестать использовать цветовую схему — отторжение.
- Считывать цвет и интерпретировать как тип — фрагильно, сломается при малейшем изменении.

**Последствия.**
- Документация продукта объясняет пользователю: цвет в нашей системе — визуальный, уникальность — в имени.
- При импорте старых моделей плагин **может** предложить мигрировать легаси-цветовую-схему в имена (пользователь подтверждает сопоставление «этот цвет = этот типоразмер»).

## ADR-013. Форм-фактор продукта — гибрид: плагин SketchUp + standalone-приложение

**Контекст.** Плагин OpenCutList у заказчика — это пример того, как сейчас устроен его workflow: 3D-модель в SketchUp, компоненты с DC и цветовой маркировкой, OCL читает и считает. Мы изучаем этот workflow как референс (см. ADR-016), но строим свой продукт. Возникает вопрос: где вообще живёт наш продукт — как SketchUp-плагин, как отдельное приложение или и то, и другое.

**Решение.** Гибрид:

- **Плагин для SketchUp** (Ruby, живёт в `%APPDATA%/SketchUp/…/Plugins/`). Роль — «датчик и исполнитель в SketchUp»:
  - чтение модели (геометрия, атрибуты, структура компонентов)
  - запись модели (вставка/замена компонентов, правка атрибутов, материалы)
  - DC-перерисовка по изменённым параметрам (см. ADR-002)
  - ImageRep-превью для feedback-loop (см. ADR-010)
  - TCP-сервер, через который с ним общается standalone-приложение (форк `mhyrr/sketchup-mcp` как основа — см. ADR-001)

- **Standalone-приложение для Windows и macOS**. Роль — «мозг продукта»:
  - UI диалога с Claude (chat-подобный, с отображением превью из SketchUp)
  - каталог сортамента (см. ADR-004), редактирование JSON с валидацией
  - генератор ТЗ в LayOut (через запуск LayOut API из SketchUp по нашему протоколу, см. секцию 04)
  - NC-конвертёр (собственный, см. ADR-017)
  - корпус примеров (см. ADR-015), управление исследовательским материалом
  - хранилище проектов и настроек (см. ADR-011 про `.nn`-сателлиты)

- **Протокол связи** плагин↔standalone: локальный TCP (как у `sketchup-mcp`), JSON-сообщения. Полноценный MCP-сервер как расширение этого протокола — см. ADR-001.

**Почему гибрид, а не один из двух.**

- Только плагин: запирает нас в ruby-среде SketchUp, UI ограничен `UI::HtmlDialog`, тяжёлые операции (NC-конвертация, генерация ТЗ по 20 страниц) блокируют SketchUp. Хранение корпуса примеров и диалогов с Claude внутри плагина — некомфортно.
- Только standalone: теряем прямой доступ к живой модели — модель надо экспортировать, обрабатывать, импортировать. DC-перерисовка невозможна без Ruby-среды SketchUp. Feedback-loop через ImageRep недоступен.

**Альтернативы.**
- «Только плагин» — отклонено по причинам выше.
- «Только standalone + свой 3D-формат» — слишком далеко от заказчика, которому нужен именно SketchUp-workflow.
- Web-приложение вместо desktop — отложено. Desktop проще: нативная работа с файлами, без серверной инфраструктуры, без регулярных подписок для заказчика.

**Последствия.**
- Разработка в два потока: Ruby-плагин и standalone-приложение (язык для standalone — см. ADR-018, если будет принят отдельно).
- Инсталлятор — один на оба компонента: standalone-инсталлятор кладёт плагин в папку SketchUp.
- Вопрос кросс-платформенности: Ruby-плагин — автоматически (SketchUp сам), standalone — надо выбирать стек.
- Публикация плагина в Extension Warehouse — опционально; может быть поставка только через сайт продукта.

## ADR-014. Бюджет геометрии детали — лёгкая, пригодная к работе на среднем ПК

**Контекст.** Мы рисуем настоящую геометрию деталей (см. ADR-002 и скорректированную формулировку задачи 22.04.2026): труба с радиусами гиба и стенкой, рез «ус 45°» на концах, отверстия под болты, скругления. Параллельно — требование: средний заказчик должен собирать модели из десятков-сотен таких деталей на обычном ноутбуке или ПК без нагруженной видеокарты. «Настоящая геометрия» не должна стать «тяжёлой геометрией».

**Решение.** Вводим явный **бюджет геометрии на деталь** и придерживаемся его в параметрическом генераторе.

Целевые значения:

| Тип детали | Бюджет faces | Бюджет edges | Сегменты радиусов | Примечание |
|---|---|---|---|---|
| Труба профильная, прямой рез | ≤ 60 | ≤ 100 | 8 на радиус гиба | Внутренний контур рисуется только при наличии обработок (отверстия, рез под углом), иначе — solid profile с внешним контуром |
| Труба профильная, ус 45° с двух сторон | ≤ 120 | ≤ 200 | 8 | Рез моделируется через обрезку плоскостью |
| Труба с 2–4 отверстиями | ≤ 200 | ≤ 320 | 8 радиус гиба, 12 окружность отверстия | Потолок для «типичной детали» |
| Лист плоский без обработок | 6 | 12 | — | Минимум из минимумов |
| Лист с контурным резом и отверстиями | ≤ 80 + 16×(число отверстий) | — | 12 на окружность | Зависит от количества обработок |
| Уголок / швеллер, прямой рез | ≤ 80 | ≤ 140 | 6–8 на внутренние радиусы | Тоньше, чем у трубы, т.к. меньше радиусов |

Контроль бюджета:
- после генерации каждой детали считаем `faces_count` и `edges_count`
- пишем в атрибут `nn_fabkit.geom_budget` значения факта и лимита
- если превышение — warning в Ruby Console и в UI, деталь помечается флагом `nn_fabkit.heavy = true`
- при открытии большой модели (> 100 таких деталей) — опциональный режим «упростить отображение» (отключение внутренних контуров на далёких деталях)

**Уровни детализации (LOD).**

Три уровня, выбираются настройкой или автоматически по расстоянию камеры:

- **LOD-0: монолитный параллелепипед** (как у заказчика сейчас, 12 edges / 6 faces). Для обзорных видов, быстрой компоновки, предварительных расчётов.
- **LOD-1: с радиусами и стенкой, без обработок** (основной режим для большинства деталей в средней модели).
- **LOD-2: полная геометрия с отверстиями, резами, фасками** (для деталей, которые идут в NC-экспорт; включается по флагу `nc_target = true` или вручную).

При NC-экспорте деталь временно повышается в LOD-2, конвертируется, возвращается в предыдущий уровень (если пользователь не зафиксировал LOD-2 явно).

**Сегментация радиусов — правило.**

- Радиус гиба профильной трубы (R = 1.5–12 мм): 8 сегментов. Визуально достаточно, для NC-конвертера — точно (IGES/STEP опираются на аналитические радиусы, а не на сегменты, см. ADR-017).
- Отверстия Ø4–Ø20: 12 сегментов. Ø20–Ø50: 16 сегментов. Крупнее — 24.
- Внешние скругления листа: 8 сегментов.

**Альтернативы.**
- «Рисуем максимально точно, оптимизируем если зависнет» — отклонено: типичная модель из 200 деталей с круглой сегментацией 24 легко уходит за 50000 faces, тормозит на старых ноутбуках.
- «Всегда LOD-0 + метаданные, реальную геометрию только при NC-экспорте» — отклонено: заказчик хочет видеть настоящие детали, в том числе для визуального контроля.

**Последствия.**
- Генератор каждого типа профиля явно описывает свой бюджет и проверяет его.
- В `06-sortament-ontology.md` добавляется поле `derived.segment_hints` (рекомендуемые сегменты для типоразмера).
- Тестовый сценарий: сборка из 200 типовых деталей не должна открываться дольше 15 секунд на референсном среднем ноутбуке (i5, 16GB, без дискретной видеокарты).

## ADR-015. Протокол корпуса примеров для обучения Claude на реальных заказах

**Контекст.** Claude не обучается в смысле «меняю веса модели». Обучение параметров происходит у Anthropic и замораживается в каждой версии модели. Но у Claude есть два канала **квази-обучения в конкретном проекте**:

1. **Project Knowledge** (эта папка `files/`) — читается при каждой сессии как контекст.
2. **Notion INDEX** (через MCP) — читается выборочно, когда Claude работает с конкретной задачей.

Если складывать в эти каналы структурированные примеры реальных заказов, Claude будет работать «как если бы он их знал» — с правильными паттернами, с узнаваемым стилем заказчика, с учётом tacit knowledge (почему такой шов, почему именно так расположили отверстия).

**Решение.** Ведём корпус примеров по строгой структуре. Каждый пример — одна папка, один реальный или учебный заказ.

Структура папки примера:

```
examples/<NN-название>/
├── model.skp                          — исходник SketchUp (копия рабочего файла)
├── model.dump.json                    — выгрузка через tools/skp_dump.rb
├── model.preview.png                  — общий вид модели, 1024×768
├── tz-svarshchiku.pdf                 — ТЗ для исполнителя (сварщика, монтажника)
├── tz-svarshchiku.md                  — тот же ТЗ в markdown (для чтения Claude)
├── nc/                                — NC-файлы, которые реально пошли на станок
│   ├── detail-001.igs
│   ├── detail-002.igs
│   └── sheet-layout.dxf
├── photos/                            — фотографии готовой детали/сборки (опц.)
│   ├── assembly-01.jpg
│   └── final.jpg
└── notes.md                           — ключевой файл, см. ниже
```

**`notes.md` — самый ценный артефакт.** Структура:

```markdown
# <Название заказа>

## Краткое описание
Что это за конструкция. Кому и для чего. Габариты.

## Материалы
- Профильная труба 40x20x2 Ст3сп — 24 м
- Лист 3 мм Ст3пс — 1 шт 1500×750
- ...

## Ключевые конструктивные решения
Почему именно эти типоразмеры (нагрузка? эстетика? доступность у поставщика?).
Почему такая схема соединений.
Где пришлось отступить от «идеального» решения и почему.

## Что уточнял исполнитель (сварщик) и что правили
Один из главных источников tacit knowledge. Переспросы сварщика — это реальные пробелы ТЗ,
которые нужно закрывать в следующих проектах.

## Что переделывали после сборки
Если что-то не сошлось при монтаже. Причины. Как в следующий раз не допустить.

## Итог
Сколько заняло. Стоимость (опц.). Удовлетворённость клиента.
```

**Объём корпуса.**

Минимум полезного: **5–8 примеров разного типа**:
- плоская рама (стол, каркас стеллажа)
- пространственная конструкция (ворота, навес, лестница)
- что-то преимущественно из листа (короб, кожух)
- что-то из круглой трубы (перила, спортивный инвентарь)
- что-то с уголком или швеллером
- мелкий быстрый заказ (полка, кронштейн) — для полноты палитры

Больше — лучше, но отдача падает. **20+ примеров** — уже избыточно для MVP; важнее качество `notes.md`, чем количество папок.

**Где хранится корпус.**

- **На первом этапе** — в Claude Project-files (`examples/`) или в папке проекта заказчика на локальной машине, синхронизируется в Notion.
- **Когда папка проекта переполняется** (заявлено заказчиком как «уже пора формировать MCP-сервер, чтобы папка files не переполнялась») — переносится в отдельное хранилище с MCP-адаптером, Claude читает через MCP-инструмент. Эта миграция — отдельное ADR, когда упрёмся в лимит.

**Процесс пополнения.**

1. Заказчик завершает реальный заказ.
2. В SketchUp открывает итоговую модель, запускает `tools/skp_dump.rb` через Ruby Console → получает `model.dump.json`.
3. Собирает папку по структуре выше. `notes.md` — пишет в свободной форме, основные разделы обязательны.
4. Помещает в корпус (через Project-files, Notion или MCP — по текущей схеме).
5. В следующей сессии Claude автоматически видит новый пример в контексте.

Темп пополнения — 1–2 примера в месяц на старте, больше по мере зрелости продукта.

**Альтернативы.**
- Fine-tune собственной модели — технически возможно, но дорого и заморозит нас на одной версии. Не оправдано для объёма данных < 100 примеров.
- RAG по всем `.skp`-файлам заказчика — слишком много шума, большинство проектов не доведены до конца или неполны.
- Ограничиться одним-двумя образцами — недостаточно для паттернизации.

**Последствия.**
- Нужен Ruby-скрипт `tools/skp_dump.rb`, превращающий `.skp` в структурированный JSON (поставляется вместе с этим ADR).
- Шаблон `notes.md` — фиксируется в `files/templates/example-notes.template.md`.
- Правило: при поступлении нового примера — обновляется INDEX в Notion (через скилл notion-librarian).

## ADR-016. OCL — только референс для проектирования, не часть интеграции (supersedes ADR-003)

**Контекст.** Первоначально (ADR-003) мы планировали интегрироваться с OpenCutList: раз заказчик им пользуется, наш плагин создаёт компоненты с материалами по OCL-конвенции, OCL читает и считает. На уточнении 22.04.2026 концепция сменилась: заказчик пользуется OCL сейчас, но в целевой картине наш продукт **полностью заменяет** ту часть workflow, где сейчас участвует OCL (спецификация, раскрой листа, подсчёт погонных метров). OCL остаётся у заказчика как отдельный инструмент, если он захочет его использовать, но наш продукт **не зависит** от него и **не подстраивается** под него.

**Решение.**

1. **Мы изучаем OCL как референс workflow.** Что именно заказчик делает через OCL сейчас, какие форматы вывода использует, что ценит, чего не хватает. Это ценный вход для проектирования собственных функций.

2. **Мы строим свою систему учёта деталей**, свою спецификацию, свой раскрой листа. Конвенция именования материалов (`"<Тип> <typesize> <марка>"`) и хранения метаданных (`nn_fabkit` attribute_dictionary) — **наши собственные решения**, а не OCL-контракт. Они могут эволюционировать независимо от того, что делает OCL.

3. **Мы не импортируем код OCL** (GPLv3-ограничение остаётся в силе, даже как референс — читать и учиться можно, копировать нельзя).

4. **Мы не пишем OCL-атрибуты в компоненты** (словарь `lairdubois_opencutlist_*`). Если заказчик хочет, чтобы наши компоненты работали и в OCL — это не наша забота, пусть настраивает OCL-материалы вручную.

5. **Мы не ограничиваем себя OCL-ограничениями.** Например: OCL требует, чтобы компонент был «solid». Мы можем использовать группы с не-solid геометрией для вспомогательных объектов (разметочные линии, осевые символы сварных швов), если это нам нужно.

**Что остаётся из старого ADR-003.**

- Схема именования материалов `"<Тип> <typesize> <марка>"` — сохраняется, но теперь как **наша конвенция**, а не OCL-требование.
- Ось X = длина — сохраняется как **наша конвенция** (совпадает с OCL-конвенцией; совпадение удобно, но мы бы её приняли в любом случае).
- Детерминированный цвет (ADR-012) — сохраняется.

**Альтернативы.**
- Оставить ADR-003 в силе: продолжать интеграцию с OCL. Отклонено — связывает нас с чужой схемой атрибутов, ограничивает свободу расширения.
- Форкнуть OCL и встроить в наш продукт под GPLv3-условием: технически возможно, но продукт должен стать GPLv3, что несовместимо с коммерческой моделью.

**Последствия.**

- Файл `05-opencutlist-integration.md` переименовывается по смыслу в «05-opencutlist-reference.md» (или снабжается заметкой о новом статусе). Материал в нём — валиден как разбор того, как устроен OCL, но не как руководство к интеграции.
- В нашем плагине **не** создаётся OCL-словарь на материалах. Если пользователь хочет — сам настроит.
- Наш раскрой листа (feature NC-модуля) строим «с нуля», без оглядки на OCL Cutting Diagrams.
- Мы можем использовать **Group** вместо **ComponentInstance** там, где это удобно (OCL игнорирует Groups, но нас это больше не ограничивает).

## ADR-017. NC-экспорт — собственный конвертёр IGES с самого начала (supersedes ADR-009)

**Контекст.** Первоначальное решение (ADR-009): в MVP оставляем SolidWorks-посредника, свой IGES-конвертёр — на этап 3+. Уточнение заказчика 22.04.2026: **SolidWorks из цепочки убираем**. Труборезы заказчика принимают IGES (`.igs`), этот формат и есть целевой выход. Модель трубореза на текущий момент не имеет значения — оба доступных станка читают IGES.

**Решение.**

1. **В MVP разрабатываем собственный конвертёр** `model → IGES`. Это критический путь продукта: без NC-экспорта заказчик не может получить файлы на станок, и весь workflow ломается.

2. **Для листового металла** — параллельно DXF. Большинство лазерно-плазменных установок для листа читают DXF, это простой 2D-формат с хорошей поддержкой в open-source. DXF реализуется первым как проверочный вариант NC-экспорта.

3. **Архитектура конвертёра.**

   - Вход: `Sketchup::ComponentDefinition` (одна деталь) или набор definitions (сборка).
   - Промежуточное представление: структура BREP на нашей стороне (faces + edges + curves + surfaces), извлечённая из SketchUp-геометрии + метаданных `nn_fabkit`.
   - Выход: IGES-файл согласно спецификации IGES 5.3 (ANSI-standard, публичная).
   - Для радиусов гиба, отверстий, резов — используем **аналитические примитивы IGES** (Type 100 — circular arc, Type 110 — line, Type 120 — surface of revolution), а не тесселяцию. Это даёт точные траектории NC независимо от сегментации отображения в SketchUp (см. ADR-014).

4. **Язык реализации.** Конвертёр живёт в standalone-приложении (см. ADR-013), а не в Ruby-плагине SketchUp. Ruby-плагин передаёт структуру детали через TCP/JSON, standalone делает конвертацию. Это обходит ограничения Ruby-среды SketchUp на производительность и зависимости, и позволяет использовать зрелые CAD-библиотеки (OpenCASCADE, или аналоги) из более подходящего языка.

5. **Библиотечная база — отдельное исследование.** Варианты:
   - OpenCASCADE (C++, LGPL) — индустриальный стандарт, полная поддержка IGES/STEP, сложный API.
   - Open CASCADE Community Edition (OCE) — форк, аналогичный.
   - `pythonocc-core` — Python-биндинги к OpenCASCADE. Подходит, если standalone пишется на Python.
   - Написать минимальный IGES-райтер вручную — работает, если ограничиться подмножеством примитивов, которые реально нужны для наших деталей (линии, дуги, цилиндры, плоскости). Спецификация IGES 5.3 ~500 страниц, но практически полезные 20% — сотня страниц.

   **Предварительный выбор для MVP** — минимальный ручной IGES-райтер на ограниченном подмножестве. Если упрёмся в сложность (нестандартные формы, свободные поверхности) — переключаемся на OpenCASCADE. Решение финализируется отдельным ADR (ADR-019 про стек standalone).

**Альтернативы.**
- Оставить ADR-009 как есть: SolidWorks-посредник. **Отклонено заказчиком.**
- Экспорт через промежуточный формат (SketchUp → DAE/STL → внешний open-source конвертер → IGES): цепочка теряет точность, управлять ошибками сложнее.
- STEP вместо IGES: технически современнее, но труборезы заказчика явно читают IGES — нет смысла менять формат в MVP. STEP добавим позже как альтернативу.

**Последствия.**
- NC-конвертёр переходит из «этап 3» в «MVP». Ресурсоёмкая задача, требует отдельного плана (как минимум 2–3 месяца работы при параллельной разработке плагина).
- На этапе проработки — нужен тестовый набор реальных деталей от заказчика (см. ADR-015) с эталонными IGES-файлами (что сейчас выдаёт SolidWorks). Сверяем наш выход с эталоном.
- В standalone-приложении резервируется модуль `nc_export/` с подмодулями `iges_writer`, `dxf_writer`, `brep_builder`.
- Тестовая инфраструктура — автоматический прогон конвертёра по набору эталонных деталей, сверка результата с эталонным IGES (текстовое diff + структурное сравнение по типам примитивов).
- Вопрос к заказчику снимается с критического: модель трубореза сейчас не важна — оба станка читают IGES. Если появится станок с другим требованием (STEP, G-code напрямую), модуль проектирования конвертёра уже заложен.
