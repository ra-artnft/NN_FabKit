# CHANGELOG — обновление базы знаний 22.04.2026

Этот архив содержит обновлённую базу знаний проекта «Интеграция Claude ↔ SketchUp / LayOut». Ниже — что изменилось относительно версии до 22.04.2026.

## Как применить

1. Разархивировать.
2. Содержимое корня архива положить поверх содержимого папки `files/` в Claude Project (с перезаписью одноимённых файлов).
3. Папку `tools/` — положить рядом, на том же уровне, что и `files/` (или внутрь `files/tools/` — как удобнее для Project Knowledge).
4. Проверить, что в новом чате Claude видит ADR-013…ADR-017 в `09-architecture-decisions.md`.

## Изменения

### 1. Namespace — утверждён `NN::FabKit` / `nn_fabkit`

Во всех файлах базы знаний заменено:
- `NN::MetalFab` → `NN::FabKit` (в Ruby-примерах)
- `nn_metalfab` → `nn_fabkit` (имя attribute_dictionary)
- `"NN MetalFab"` → `"NN FabKit"` (display name расширения)
- `NnMetalFab::...` → `NnFabKit::...` (в переходных примерах)

Причина: «NN MetalFab» относилось к металл-ветке первой итерации; после уточнения задачи продукт стал зонтичным (металл + мебель). `FabKit` — нейтральное placeholder-имя. Может быть изменено при утверждении финального названия продукта — одной заменой в будущем.

### 2. ADR-003 → помечен SUPERSEDED, заменён ADR-016

**Было:** OCL — целевая интеграция. Наш плагин создаёт компоненты с материалами по OCL-конвенции, OCL читает и считает.

**Стало (ADR-016):** OCL — только референс для проектирования. Наш продукт **полностью заменяет** функции OCL (спецификация, раскрой листа, подсчёт материала). Мы **не пишем** OCL-атрибуты в компоненты, **не зависим** от его конвенций, **не включаем** код. Конвенция именования материалов и ось X = длина — сохранены, но уже как **наши** решения (совпадение с OCL удобно, но не обязательно).

Следствие для `05-opencutlist-integration.md`: добавлена шапка со сменой статуса. Основной текст файла сохранён как исторический разбор того, как устроен OCL.

### 3. ADR-009 → помечен SUPERSEDED, заменён ADR-017

**Было:** В MVP IGES получаем через посредника (SolidWorks у заказчика). Свой конвертёр — на этап 3+.

**Стало (ADR-017):** Собственный конвертёр model → IGES входит в MVP. SolidWorks-посредник убирается из цепочки с самого начала. Архитектура — конвертация живёт в standalone-приложении (не в Ruby-плагине), использует аналитические примитивы IGES 5.3. Для MVP — минимальный ручной IGES-райтер, при упоре в сложность — переход на OpenCASCADE.

Модель трубореза больше не является критическим открытым вопросом: оба станка заказчика читают IGES.

### 4. Новый ADR-013 — гибридный форм-фактор

Продукт состоит из двух компонентов:
- **Плагин SketchUp** (Ruby): читает/пишет модель, делает DC-перерисовку, рендерит превью через ImageRep, держит TCP-сервер.
- **Standalone-приложение** (Windows + macOS): UI диалога с Claude, каталог сортамента, генератор ТЗ в LayOut, NC-конвертёр, корпус примеров, хранилище проектов.

Связь — локальный TCP / JSON (форк `sketchup-mcp` как стартовая точка).

### 5. Новый ADR-014 — бюджет геометрии детали

Параметрический генератор рисует настоящую геометрию (радиусы гиба, стенки, резы, отверстия), но с явным бюджетом faces/edges и тремя LOD-уровнями (монолитный параллелепипед → с радиусами → с обработками). Цель — сборка из 200 типовых деталей должна открываться не дольше 15 секунд на среднем ноутбуке без дискретной видеокарты.

### 6. Новый ADR-015 — протокол корпуса примеров

Структура папки примера (`model.skp` + `model.dump.json` + `tz-svarshchiku.md/pdf` + NC-файлы + фотографии + `notes.md`). Минимум полезного корпуса — 5–8 примеров разного типа. Главный артефакт — `notes.md` с tacit knowledge (что переспрашивал сварщик, что переделывали после сборки).

Claude **не обучается** в смысле «меняю веса». Корпус примеров в Project Knowledge / Notion / MCP — это квази-обучение через контекст: Claude читает примеры каждую сессию.

### 7. Новый `tools/skp_dump.rb`

Ruby-скрипт, копипастится в Ruby Console в SketchUp. Выгружает текущую модель в `<basename>.dump.json` рядом с `.skp`-файлом. Выход — структурированный JSON со всеми definitions, instances, materials, layers, pages, атрибутами, геометрией (в миллиметрах, точность 3 знака).

Применение: пополнение корпуса примеров (ADR-015).

**Важно:** скрипт не проверен синтаксически вне SketchUp (в среде сборки Ruby нет). Перед использованием — прогнать через Ruby Console, при ошибке синтаксиса SketchUp укажет строку.

## Открытые вопросы — обновлено

**Закрыто:**
- Модель трубореза — неактуально (оба станка читают IGES, ADR-017).
- Форм-фактор продукта — гибрид (ADR-013).
- Namespace — `NN::FabKit` / `nn_fabkit`.

**Переформулировано:**
- Бюджет геометрии — теперь есть (ADR-014), остаётся валидация на реальных моделях.
- NC-конвертёр — из «когда-нибудь в этап 3+» перешёл в MVP, требует отдельного плана реализации.

**Новое:**
- ADR-018 (будущий): стек для standalone-приложения (Python? Rust? Electron? Tauri?). Блокирует старт разработки standalone.
- ADR-019 (будущий): библиотечная база для IGES (ручной райтер / OpenCASCADE / pythonocc). Блокирует финализацию NC-конвертёра.
- Когда и как переносить корпус примеров в MCP-сервер (заявлено заказчиком, но отдельный ADR ещё не написан — пишется при подходе к лимиту Project Knowledge).
