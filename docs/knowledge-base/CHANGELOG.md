# CHANGELOG — обновление базы знаний 26.04.2026

## [Unreleased] / Added — 2026-04-26

- `11-gost-profile-tubes-radii.md` — справочник по ГОСТам на профильные трубы
  и радиусам скругления для feature recognition в CypTube. Покрывает
  ГОСТ 30245-2003, ГОСТ 30245-2012, ГОСТ 8645-68, ГОСТ 8639-82,
  ГОСТ 13663-86, ГОСТ 32931-2015. Сравнительная таблица + формулы
  R = f(t) + допуски + правила inner radius для surface-моделирования.

**Ключевое расхождение, обнаруженное при подготовке справочника:** все
62 записи в `gost-30245-rect-tube.json` содержат `outer_radius_mm`,
рассчитанный по формуле `R = 1,5 × t`, что соответствует *нижней
границе допуска* ГОСТ 30245-2003 и опции «по соглашению» в ГОСТ
8639/8645 — но **не номиналу**. Номинальная формула ГОСТ 30245-2003
(п. 3.5): `R = 2,0×t` при t ≤ 6 мм, `R = 2,5×t` при 6 < t ≤ 10 мм,
`R = 3,0×t` при t > 10 мм. Реальные прайсы российских поставщиков
(engineerum.com, elsort.ru, as-tim.ru, metallobazav.ru) единогласно
дают R = 2,0×t для типовых сечений (60×40×3 → R=6,0 мм; 100×100×4 →
R=8,0 мм; 120×120×5 → R=10,0 мм). Текущий JSON занижает радиусы на
25 % относительно стандарта и фактической поставки, что критично для
feature recognition в CypTube/TubesT и должно быть исправлено
отдельной миграцией каталога. Также правило, прописанное в
`06-sortament-ontology.md` («R = 1,5×t для t ≤ 6 мм, R = 2,0×t для
t > 6 мм»), нужно переписать на ГОСТ-формулу.

# CHANGELOG — обновление базы знаний 25.04.2026

## Добавлен `10-iges-for-tube-nc.md` — справочник по IGES для трубного NC

Новый раздел базы знаний, ориентированный на разработку собственного
IGES-конвертёра (ADR-017). Содержит:

- **Раздел 1.** IGES Entity Types для трубного NC — Type 110, 100, 102,
  120, 122, 142, 144, 186 и топологические сущности 502/504/508/510/514.
  Параметры в P-section, ASCII-фрагменты, сравнение представлений
  (wireframe vs surface vs trimmed surface vs BREP).
- **Раздел 2.** Что читают трубопрофилерезы рынка РФ/СНГ — таблица
  по 11 моделям (BLM, TRUMPF, Mazak, Bystronic, Han's Laser, Bodor,
  Golden Laser, HGG, Vernon, Prima Power, Bochu/FSCUT). Поддерживаемое
  подмножество IGES, известные несовместимости. Косвенный анализ стека
  заказчика-0 (предположительно FSCUT-семейство → TubesT/TubePro).
- **Раздел 3.** Практика генерации IGES для труб — ориентация (Z vs X),
  outer/inner contour, единицы (UNIT_FLAG=2 / "MM"), точность
  (1e-6 mm), формат чисел (`%.6f`, без E-нотации), порядок сборки
  файла снизу вверх.
- **Раздел 4.** Минимальное подмножество — 30 сущностей для surface
  modeling одной трубы. Stub-фрагмент IGES для 40×20×2 L=600 мм с
  правильной структурой S/G/D/P/T.
- **Раздел 5.** Альтернативы IGES — STEP AP203/214/242, DSTV NC1
  (с таблицей кодов профилей и операций), DXF 3D, ACIS SAT, Parasolid
  X_T. Когда что предлагать.
- **Раздел 6.** Библиотеки и инструменты — pythonocc-core/OpenCASCADE,
  roseengineering/igeswrite, pyvista/pyiges, FreeCAD, CAD Assistant.
  Сравнение языков для standalone (Python / Rust / Go / C# / C++).
- **Раздел 7.** ~50 ссылок на спецификации, документацию станков,
  GitHub-репо, форумы.
- **Раздел 8.** Рекомендации для NN FabKit — точное число entities (48)
  для нашего LOD-1, рекомендация Python+custom writer для MVP, конкретные
  значения G-section параметров, чек-лист «ready for first machine test»,
  список открытых вопросов для эмпирической проверки.

**Ключевые выводы, влияющие на разработку:**

1. Целевой rendering — **surface model на Type 144** без BREP. BREP
   (Type 186) «poorly implemented» большинством CAM-систем
   ([cadinterop.com](https://www.cadinterop.com/en/formats/neutral-format/iges.html)),
   и заказчик-0 предположительно работает на FSCUT/TubesT, который
   BREP игнорирует.
2. Текущий wireframe.rb (Type 110 + 100, 40 entities) — годится только
   для visual sanity-check. Для NC нужно расширение до ~48 entities с
   Type 102 + 122 + 142 + 144 + 190.
3. Язык standalone — **Python с собственным writer'ом** для MVP (~600-800
   строк кода), c опциональным fallback на pythonocc-core если упрёмся
   в сложность. Это закрывает открытый ADR-019.
4. Координаты, единицы — мм всегда; ось длины — Z; точность 1 нм
   (`%.6f`); newline `\r\n`; ASCII без BOM; IGES 5.3 (version_flag=11).
5. Composite curves (Type 102) — потенциальная проблема для китайских
   систем; план B — писать каждую кривую как отдельный trim-target.

# CHANGELOG — обновление базы знаний 24.04.2026 (часть 2)

## ADR-018 добавлен — MCP-мост собственной реализации (supersedes ADR-001)

`mhyrr/sketchup-mcp`, на который опирался ADR-001 как «MIT-licensed base», на самом деле не имеет лицензии (`gh api .../sketchup-mcp` возвращает `license: null`, файла LICENSE в репо нет). По умолчанию это all-rights-reserved — форкать или копировать код нельзя.

ADR-018 фиксирует pivot: пишем собственный минимальный TCP-мост + Python MCP server в монорепо NN FabKit. Архитектурные паттерны (TCP/JSON-RPC/timer polling) — общеизвестные, не защищены копирайтом. Код mhyrr не используем, только публично описанные идеи из README.

ADR-001 помечен как superseded ADR-018. План реализации — `docs/specs/spec-02-mcp-bridge.md`.

# CHANGELOG — обновление базы знаний 22.04.2026

Этот архив содержит обновлённую базу знаний проекта «Интеграция Claude ↔ SketchUp / LayOut». Ниже — что изменилось относительно версии до 22.04.2026.

## Как применить

1. Разархивировать.
2. Содержимое корня архива положить поверх содержимого папки `files/` в Claude Project (с перезаписью одноимённых файлов).
3. Папку `tools/` — положить рядом, на том же уровне, что и `files/` (или внутрь `files/tools/` — как удобнее для Project Knowledge).
4. Проверить, что в новом чате Claude видит ADR-013…ADR-017 в `09-architecture-decisions.md`.

## Изменения

### 1. Namespace — утверждён `NN::FabKit` / `nn_fabkit`

Во всех файлах базы знаний заменено:
- `NN::MetalFab` → `NN::FabKit` (в Ruby-примерах)
- `nn_metalfab` → `nn_fabkit` (имя attribute_dictionary)
- `"NN MetalFab"` → `"NN FabKit"` (display name расширения)
- `NnMetalFab::...` → `NnFabKit::...` (в переходных примерах)

Причина: «NN MetalFab» относилось к металл-ветке первой итерации; после уточнения задачи продукт стал зонтичным (металл + мебель). `FabKit` — нейтральное placeholder-имя. Может быть изменено при утверждении финального названия продукта — одной заменой в будущем.

### 2. ADR-003 → помечен SUPERSEDED, заменён ADR-016

**Было:** OCL — целевая интеграция. Наш плагин создаёт компоненты с материалами по OCL-конвенции, OCL читает и считает.

**Стало (ADR-016):** OCL — только референс для проектирования. Наш продукт **полностью заменяет** функции OCL (спецификация, раскрой листа, подсчёт материала). Мы **не пишем** OCL-атрибуты в компоненты, **не зависим** от его конвенций, **не включаем** код. Конвенция именования материалов и ось X = длина — сохранены, но уже как **наши** решения (совпадение с OCL удобно, но не обязательно).

Следствие для `05-opencutlist-integration.md`: добавлена шапка со сменой статуса. Основной текст файла сохранён как исторический разбор того, как устроен OCL.

### 3. ADR-009 → помечен SUPERSEDED, заменён ADR-017

**Было:** В MVP IGES получаем через посредника (SolidWorks у заказчика). Свой конвертёр — на этап 3+.

**Стало (ADR-017):** Собственный конвертёр model → IGES входит в MVP. SolidWorks-посредник убирается из цепочки с самого начала. Архитектура — конвертация живёт в standalone-приложении (не в Ruby-плагине), использует аналитические примитивы IGES 5.3. Для MVP — минимальный ручной IGES-райтер, при упоре в сложность — переход на OpenCASCADE.

Модель трубореза больше не является критическим открытым вопросом: оба станка заказчика читают IGES.

### 4. Новый ADR-013 — гибридный форм-фактор

Продукт состоит из двух компонентов:
- **Плагин SketchUp** (Ruby): читает/пишет модель, делает DC-перерисовку, рендерит превью через ImageRep, держит TCP-сервер.
- **Standalone-приложение** (Windows + macOS): UI диалога с Claude, каталог сортамента, генератор ТЗ в LayOut, NC-конвертёр, корпус примеров, хранилище проектов.

Связь — локальный TCP / JSON (форк `sketchup-mcp` как стартовая точка).

### 5. Новый ADR-014 — бюджет геометрии детали

Параметрический генератор рисует настоящую геометрию (радиусы гиба, стенки, резы, отверстия), но с явным бюджетом faces/edges и тремя LOD-уровнями (монолитный параллелепипед → с радиусами → с обработками). Цель — сборка из 200 типовых деталей должна открываться не дольше 15 секунд на среднем ноутбуке без дискретной видеокарты.

### 6. Новый ADR-015 — протокол корпуса примеров

Структура папки примера (`model.skp` + `model.dump.json` + `tz-svarshchiku.md/pdf` + NC-файлы + фотографии + `notes.md`). Минимум полезного корпуса — 5–8 примеров разного типа. Главный артефакт — `notes.md` с tacit knowledge (что переспрашивал сварщик, что переделывали после сборки).

Claude **не обучается** в смысле «меняю веса». Корпус примеров в Project Knowledge / Notion / MCP — это квази-обучение через контекст: Claude читает примеры каждую сессию.

### 7. Новый `tools/skp_dump.rb`

Ruby-скрипт, копипастится в Ruby Console в SketchUp. Выгружает текущую модель в `<basename>.dump.json` рядом с `.skp`-файлом. Выход — структурированный JSON со всеми definitions, instances, materials, layers, pages, атрибутами, геометрией (в миллиметрах, точность 3 знака).

Применение: пополнение корпуса примеров (ADR-015).

**Важно:** скрипт не проверен синтаксически вне SketchUp (в среде сборки Ruby нет). Перед использованием — прогнать через Ruby Console, при ошибке синтаксиса SketchUp укажет строку.

## Открытые вопросы — обновлено

**Закрыто:**
- Модель трубореза — неактуально (оба станка читают IGES, ADR-017).
- Форм-фактор продукта — гибрид (ADR-013).
- Namespace — `NN::FabKit` / `nn_fabkit`.

**Переформулировано:**
- Бюджет геометрии — теперь есть (ADR-014), остаётся валидация на реальных моделях.
- NC-конвертёр — из «когда-нибудь в этап 3+» перешёл в MVP, требует отдельного плана реализации.

**Новое:**
- ADR-018 (будущий): стек для standalone-приложения (Python? Rust? Electron? Tauri?). Блокирует старт разработки standalone.
- ADR-019 (будущий): библиотечная база для IGES (ручной райтер / OpenCASCADE / pythonocc). Блокирует финализацию NC-конвертёра.
- Когда и как переносить корпус примеров в MCP-сервер (заявлено заказчиком, но отдельный ADR ещё не написан — пишется при подходе к лимиту Project Knowledge).
